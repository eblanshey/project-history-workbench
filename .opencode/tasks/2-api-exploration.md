## Goal
Explore the FreeCAD Python API to understand document structure, object properties, and hierarchy before designing domain objects for snapshots and diffing logic.

## Context

## Decisions Made
| Decision | Rationale | Alternatives Considered |
|----------|-----------|------------------------|
| API exploration BEFORE domain design | Ensures domain objects reflect actual FreeCAD data structures, not assumptions | Designing domain first could lead to refactoring after discovery |
| Use existing BasicFile.FCStd as test fixture | Avoids creating new test data; uses what's already available | Creating a "golden test document" with specific known values |
| Output complete raw API responses | Provides full context for edge cases and property types | Summarized output might miss important details |
| **Tiered filtering with hard-coded defaults** | MVP approach: exclude known noise properties, include everything else | Dynamic configuration via UI (deferred to future phase) |
| **Defer Shape/material serialization** | Complex data not needed for initial MVP; focus on parametric properties | Include in v1 (pushed to future phase) |
| **No special origin detection logic** | Identical origins produce no diff; let diff engine handle it | Custom logic to detect/filter origin groups |
| **Merged tree with DiffState** | Single tree view is cleaner and more intuitive for users | Two-column side-by-side comparison |

## Notes & Findings

### Key API Findings

#### Expression Support
- **NO `getExpression()` method** on any object types in this FreeCAD version
- Expressions stored in `ExpressionEngine` list as `[(property_name, expression_string), ...]`
- Must parse ExpressionEngine to get expression for a specific property

```python
def get_expression(obj, prop_name):
    for entry in getattr(obj, 'ExpressionEngine', []):
        if isinstance(entry, (list, tuple)) and len(entry) >= 2:
            if entry[0] == prop_name:
                return entry[1]
    return None
```

#### Property Types Discovered
| Python Type | Count | Example | Serialization |
|-------------|-------|---------|---------------|
| `str` | 99 | "Pad_Main" | JSON string |
| `bool` | 48 | True/False | JSON boolean |
| `list` | 47 | `[<object>, <object>]` | JSON array |
| `Placement` | 24 | Position + Rotation | Custom format |
| `Quantity` | 13 | "10.0 mm" | `{value, unit}` |
| `NoneType` | 7 | None | null |
| `float` | 4 | 1e-06 | JSON number |
| `Vector` | 2 | (x, y, z) | `[x, y, z]` |

#### Hierarchy Methods
- `obj.InList` - Objects that reference this object (parents)
- `obj.OutList` - Objects referenced by this object (children)
- `obj.getSubObjects()` - Returns tuple of subobject name strings

## Post-Exploration: Domain Object Design Questions Answered

### Final Domain Model Structure

#### Core Types
```python
# placement.py
@dataclass(frozen=True)
class Vector:
    x: float
    y: float
    z: float

@dataclass(frozen=True)
class Rotation:
    axis: Vector
    angle: float  # radians

@dataclass(frozen=True)
class Placement:
    base: Vector
    rotation: Rotation

# property_value.py
@dataclass(frozen=True)
class PropertyValue:
    value: object          # The actual serialized property value
    type_name: str         # Python type or FreeCAD type (e.g., "Quantity", "Placement")
    expression: str | None # Expression from ExpressionEngine parsing (if any)

# snapshot.py
@dataclass(frozen=True)
class TreeNode:
    name: str              # obj.Name (unique identifier)
    type_id: str           # obj.TypeId (e.g., "PartDesign::Pad")
    label: str             # obj.Label (user-friendly name)
    placement: Placement | None  # Position/orientation if available
    properties: dict[str, PropertyValue]
    children: list['TreeNode']  # From OutList

@dataclass(frozen=True)
class Snapshot:
    name: str
    timestamp: datetime
    root: TreeNode
```

#### Diff Types (Merged Tree Approach)
```python
# diff_result.py
from enum import Enum

class DiffState(Enum):
    ADDED = "added"        # New on right (green highlight)
    DELETED = "deleted"    # Removed from left (strikethrough/red)
    MODIFIED = "modified"  # Properties changed (blue highlight)
    UNCHANGED = "unchanged"  # No changes (normal)

@dataclass(frozen=True)
class PropertyDiff:
    property_name: str
    old_value: PropertyValue
    new_value: PropertyValue
    changed_expression: bool

@dataclass(frozen=True)
class NodeDiff:
    """Merged tree node with change state."""
    name: str
    type_id: str
    state: DiffState
    property_diffs: list[PropertyDiff]  # Only populated if MODIFIED
    children: list['NodeDiff']

@dataclass(frozen=True)
class DiffSummary:
    added_count: int
    deleted_count: int
    modified_count: int
    unchanged_count: int

@dataclass(frozen=True)
class DiffResult:
    """Single merged tree for UI rendering."""
    summary: DiffSummary
    root: NodeDiff  # Merged tree structure for direct rendering
```

### UI Rendering Approach

The merged `DiffResult.root` tree is rendered directly with CSS styling based on `DiffState`:

```css
.node-added { color: green; font-weight: bold; }
.node-deleted { text-decoration: line-through; color: red; }
.node-modified { color: blue; }
.node-unchanged { }
```

For MODIFIED nodes, property diffs are shown inline:
```
Pad_Main (modified)
  - Length: 10.0 mm → 15.0 mm
  - Expression: (none) → <<MyVarSet>>.TargetLength
```

### 1. TreeNode structure
- Uses merged approach with `DiffState` for UI highlighting
- No parent references needed (tree navigation is top-down only)
- Placement extracted as its own dataclass

### 2. PropertyValue structure
- Includes `expression` field parsed from ExpressionEngine
- Tracks both value AND expression separately
- Type name helps with custom comparators

### 3. Diff strategy
- **Merged tree**: Single tree with state, not two-column comparison
- **Value comparison**: Compare serialized values for equality
- **Expression tracking**: Track expression changes separately via `changed_expression` flag
- **Type handling**: Handle Placement, Quantity, Vector with custom comparators
- **Exclusions**: Apply AUTO_EXCLUDED_PROPERTIES before comparison

### 4. Exclusions
- **Properties**: Hard-coded list of auto-generated properties
- **Types**: Start with empty list (permissive), let diff engine find differences
- **Future**: User-configurable via SettingsPort
